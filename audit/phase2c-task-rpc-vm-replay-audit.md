## Phase 2C Summary

The web restores the CLIENT-side VM completely (boot, CBOR feed, notifyUI loop, ui_emu interaction state machine, `__fk` data/filter/log bridges, ResetClientLua), which is what enables in-browser interactive play against the external asio server. The SERVER-side scheduler/task/RPC/CLI run in asio and are not reimplemented in the browser/gateway.

**Status:** successfully restored 8 (P2C-006, 012, 014, 016, 018, 019, 020; plus client side of 002) · simplified restoration 5 (002, 007, 009, 015, 017) · restored incorrectly 1 (013, copied-Lua drift) · not restored 6 (001, 003, 004, 005, 010, 011).

**Key changes since the M2-era report:** new rows P2C-016..020 document the now-complete ui_emu interaction bridge, `__fk` filter/data/log bridges, and CBOR feed — none of which the M2 report covered. Replay/record (P2C-010/011) and server scheduler/task/RPC/CLI (P2C-001/003/004/005) remain not restored by design (asio owns the server).

**Correction note:** an earlier draft of P2C-017 referenced a nonexistent `apps/web/src/table/RequestUI/*.tsx` directory; the actual request UI lives in `RequestPopup.tsx` + `popupStore.ts`. Corrected here.
