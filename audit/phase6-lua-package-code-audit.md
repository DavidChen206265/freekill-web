## Phase 6 Summary

The web project restores the client Lua engine by running a forked public Lua tree inside wasmoon, with JS natives + a Lua `CppClient` shim. M5 extended this with extension packs (utility/standard_ex/sp) mounted alongside freekill-core, increasing bundled generals/skills. The production game RULES still run server-side on asio; the web Lua is the client (display/ui_emu + skill definitions for client rendering).

Successful/adaptive restorations: package loading, disabled-pack handling, game-mode feasibility, dynamic `Card:getSkill(player)`, special-skill use/response, stable move logs, room RNG fallbacks, added effect data, and (M4-I) MoveCardInBoard popup. Parity risks: missing `compat/askfor.lua` (legacy `askFor*` callers fail), behavior-changing forks (`hujia`/`jizhi`/`distribution_select_skill`/`game_rule`/`eight_diagram`), incomplete React consumption of `Interaction`, and generic AskForPoxi UI.

Highest-risk functional gaps:
- Legacy extensions using `Room:askFor*` (from `compat/askfor.lua`) would fail.
- `AskForPoxi` reaches React as a generic picker without poxi prompt/filter/feasible (Phase 5 P5-022).
- Forked package rule files may change trigger/target legality; need scenario tests.
- Public Lua + extension packs are forked runtime copies; source updates don't auto-propagate.
- Extension-pack MD5 mismatch (asio enabled set ≠ web EXTENSION_PACKS) causes gateway rejection.
